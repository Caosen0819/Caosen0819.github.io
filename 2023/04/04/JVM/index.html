JVM

04 Apr 2023 in Java

1、Java内存区域

主要有五大部分：程序计数器（私有）、虚拟机栈（私有）、本地方法栈（私有）、堆（公有）、方法区（公有），不过在8之后，方法区就已经被元空间代替了，而元空间在本地内存。

1、程序计数器

当前线程执行的字节码的行号指示器，什么循环、分支、线程恢复都靠这个。

2、虚拟机栈

每个方法调用的时候都会生成一个栈帧，栈帧里面存储了局部变量表，操作数栈、动态连接、方法出口等。

局部变量表里面放了方法参数、方法内部定义的局部变量。包括各类基本数据类型、对象引用(reference) ，以及returnAddress类型。

对象引用要完成两件事情：找到在JVM堆里面的对象，还有一个是找到元空间存储的类型信息。

操作数栈：保存计算过程的中间结果,同时作为计算过程中变量临时的存储空间

动态连接：每一个栈帧内部都包含一个指向运行时常量池中该栈帧所属方法的引用

方法出口：存放调用该方法的pc寄存器的值

3、本地方法栈

和虚拟机栈很像，区别是为了虚拟机使用本地native方法服务的。

4、堆

虚拟机管理的内存中最大的一块，几乎所有的对象实例都在这里。这里有一块是字符串常量池，存的是堆String实例的引用。

5、方法区

存储被虚拟机加载的类型信息、常量、静态变量，编译后的代码缓存。不过在8之后就移到元空间了。

运行时常量池：

Class文件除了类的版本字段方法接口描述外，还有一个常量池，用于存放编译期间生成的字面量和符号引用，而当类加载到内存中后，jvm把符号引用换成直接引用，将class常量池中的内容存放到运行时常量池中。

对象的创建

1、当遇到new，先去检查这个指令的参数能不能再常量池找到类的符号引用，然后去检查是不是已经类加载、解析、初始化。

2、如果没有，就先执行类加载。

3、然后分配内存，内存的大小在类加载的时候就确认了。具体分配用“指针碰撞”还是“空闲列表”就看用那种垃圾收集器。为了保证不出现并发问题：有两种方案。1）CAS+失败重试。2）给线程自己预先分配本地线程分配缓冲。（TLAB）

每一个对象在堆里面分为三块：对象头、实例数据、对其填充。

  • markword就是记录哈希码、gc分代、锁标志、锁id，偏向锁。
  • 类型指针就是指向类型元数据的指针

对象的访问定位

1、句柄方法。reference放的是存储的是句柄地址，句柄包含对象实例数据和类型数据，各自的具体位置。句柄最大的优势就是即使在垃圾收集的时候对象被移动了，只用修改句柄中的指针就好，不用修改reference。

2、直接指针。存储的就是对象地址。他的优势是少一次间接访问的消耗。

2、内存溢出异常

除了程序计数器之外，其他几个运行时区域都有可能发生oom异常。

1、堆溢出oom

不断创建对象，然后避免垃圾回收

2、虚拟机栈和本地方法栈溢出sof

栈的深度超过最大深度，或者栈的容量无法申请到内存

3、方法区和运行时常量池溢出oom

3、字节码文件

  • 魔数：唯一的作用就是确定这个文件是否为一个虚拟机可以接受的Class文件
  • 次版本号
  • 主版本号
  • 常量池：第一个表类型的数据
  • 主要存放两大类常量：字面量和符号引用。比如包名、方法名接口名、变量名等等，但是Class常量池中他就是一个符号，需要在类加载的时候翻译成具体的内存地址。
  • 访问标志：有16个标志位，是类还是接口、是否public 是否abstract 是否final
  • 类索引、父类索引、接口索引：用着三项去确认继承关系
  • 字段表集合：描述类中声明的变量。最主要有三个部分：access_flag，是否public、是否provate、是否static。name_index：常量池的第几个，比如0005，代表常量池的第五个值（m）是discriptor_index 是int类型（0006：第六个值I，代表是int类型）
  • 方法表：方法表的结构和字段表基本一样，不过是针对方法的
  • 属性表：太复杂了

4、类加载机制

类加载就是把描述类的数据从Class文件里面加载到内存，并对其进行校验、转换、解析、初始化，最终变成可以被虚拟机直接使用的Java类的原型-类模板对象。

类加载分成三个阶段，加载、链接、初始化。连接可以分成，验证、准备、解析

1、加载

加载是类加载过程的第一步，主要完成下面 3 件事情：

  1. 通过全类名获取定义此类的二进制字节流
  2. 将字节流所代表的静态存储结构转换为方法区的运行时数据结构（JDK 1.8之前：永久代；JDK 1.8之后：元空间）
  3. 在内存中生成一个代表该类的 Class 对象，作为方法区这些数据的访问入口（堆）

2、验证

确保Class文件的信息复合虚拟机的约束

1、文件格式验证。就是去检查类文件格式是否规范。什么魔数主次版本常量池字段池方法池

2、元数据验证。语义分析，有没有父类，有没有继承不能继承的父类。

3、字节码校验。最复杂的阶段。通过数据流分析和控制流分析程序语义是否合法，主要就是检查方法体

4、符号引用验证。符号引用验证发生在解析阶段，就是把符号引用转化为直接引用的时候去验证的。

3、准备阶段

为静态变量分配内存，并设置初始值的阶段。这里如果是通常阶段，这里的初始值是默认的，不是你定义的什么1,2,3这类，实则是0。如果用final修饰了，那就是你自己定义的了。

4、解析

把常量池中的符号引用转化为直接引用的阶段。主要针对类和接口，字段，类方法。我们拿方法举个例子，我们执行方法的时候需要知道方法在哪里，而类文件就准备了一个方法表。所以我们只需要该方法在方法表中的偏移量就可以直接调用该方法了。不过方法表中，有一部分方法调用可以有一个确定，这个叫解析。但是有另外一部分，如果 Human h = new Man。这种可能方法调用不能确定，这里就解析不了。能解析的叫非虚方法，不能解析的叫虚方法。虚方法静态分配实现重载（静态类型），动态分配实现重写（invokevirtual）。

解析的步骤就是：

在类C中查找有没有与目标匹配的简单名称和表舒服。如果有就直接返回。

否则，就在类C的父类中递归查找自下而上。

否则，在类C的接口中基付接口查找是否有匹配的。

否则失败。

相同的符号引用不同的JVM机器上对应的直接引用可能不同，直接引用一般对应已加载嗲内存中的一个具体对象

5、初始化

初始化就是执行类构造器的过程，而这个类构造器做的事情就是对类中所有类变量的赋值操作和所有静态语句快的复制操作。

顺序就是语句在源文件的顺序

5、类加载器

类加载器的作用就是实现类加载过程中的加载这一步。每一个类都有一个引用指向类加载器，Java一直保持着三层类加载器、双亲委派的类加载器架构。

三层：

启动类加载器：负责加载JAVA_HOME/LIB目录的或者被-xbootclasspath指定。

扩展类加载器：负责加载JAVA_HOME/LIB/EXT的类或者被JAVA.EXT.DIRS系统变量制定的所有的类库。由于是由Java代码实现，所以开发者可以在程序中直接使用这个加载器去加载Class文件

应用程序类加载器：负责加载用户类路径CLasspath上所有的类库。开发者也可以直接使用这个

双亲委派：

双亲委派是类加载器的组织模型。

在类加载的时候，系统会首先判断当前类是否被加载过。已经被加载的类会直接返回，否则才会尝试加载（每个父类加载器都会走一遍这个流程）。

类加载器在进行类加载的时候，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成（调用父加载器 loadClass()方法来加载类）。这样的话，所有的请求最终都会传送到顶层的启动类加载器 BootstrapClassLoader 中。

只有当父加载器反馈自己无法完成这个加载请求（它的搜索范围中没有找到所需的类）时，子加载器才会尝试自己去加载（调用自己的 findClass() 方法来加载类）。

如果子类加载器也无法加载这个类，那么它会抛出一个 ClassNotFoundException 异常。

好处

可以避免类的重复加载（JVM 区分不同类的方式不仅仅根据类名，相同的类文件被不同的类加载器加载产生的是两个不同的类），也保证了 Java 的核心 API 不被篡改。

打破双亲委派

1、自定义类加载器，重写 loadClass()方法之后，我们就可以改变传统双亲委派模型的执行流程。例如，子类加载器可以在委派给父类加载器之前，先自己尝试加载这个类，或者在父类加载器返回之后，再尝试从其他地方加载这个类。具体的规则由我们自己实现，根据项目需求定制化。

解决方案：新添加一个findClass

2、高层的类加载器（SharedClassLoader）借助子类加载器（ WebAppClassLoader）来加载业务类

解决方案：线程上下文类加载器。 线程上下文类加载器的原理是将一个类加载器保存在线程私有数据里，跟线程绑定，然后在需要的时候取出来使用。这个类加载器通常是由应用程序或者容器（如 Tomcat）设置的。

3、OSGI实现模块化热部署，把类加载从双亲委派变成更加复杂的网状结构。

模块化

什么是JDK9模块化系统

  • 是一种新的组织方式，将Java SE分成多个模块，每个模块都有自己的API和实现
  • 每个模块都有一个唯一的标识符和版本号，可以独立地进行开发

为了模块化系统的顺利实施。

扩展类加载器被平台类加载器取代。加载器用于加载 JDK 中的模块

双亲委派机制也发生了变动。

平台或应用程序类加载器收到类加载请求后，在委派给父加载器前，先判断该类是否能够归属到某一个模块中，如果可以，就先找到负责这个模块的加载器进行加载。每个模块都有一个类加载器
