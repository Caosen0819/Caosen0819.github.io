Redis复习

Oct 09, 2023 in Java

1、Redis数据结构

2、Redis线程模型

1、Redis是不是单线程的？

1）以前严格意义上来说也不是单线程，而是负责处理请求的线程是单线程（获取 (socket 读)、解析、执行、内容返回 (socket 写)），还有后台线程
2）6.0.x后，由于网络的瓶颈出现，Redis采用了多线程来解决这个问题

流程简述如下：

  • 主线程获取 socket 放入等待列表
  • 将 socket 分配给各个 IO 线程（并不会等列表满）
  • 主线程阻塞等待 IO 线程(多线程)读取 socket 完毕
  • 主线程执行命令 - 单线程（如果命令没有接收完毕，会等 IO 下次继续）
  • 主线程阻塞等待 IO 线程(多线程)将数据回写 socket 完毕（一次没写完，会等下次再写）

在Redis6.0中新增加了多线程的功能来提高I/O读写性能，他的主要实现思路是将主线程的IO读写任务拆分给一组独立的线程去执行，这样就可以使多个socket的读写并行化了，配合I/O多路复用技术可以让单个线程高效处理多个连接请求（尽量减少网络IO的时间消耗），将最耗时的socket读取，请求解析，写入单独外包出去，剩下的命令执行任然是由主线程串行执行和内存的数据交互。

2、单线程是怎么样子的

其实redis的单线程是先初始化后一直走一个循环处理。初始化的时候就做了三件事情：

  • 创建一个 epoll 对象，就是我们的io多路复用的实现
  • 对配置的监听端口进行 listen
  • 把 listen socket 让 epoll 给管理起来，注册accept处理函数。

做完这些之前之后就要真正处理用户请求，他会调用 epoll_wait 函数等待事件的到来，交给事件分发器。

  • 若发现 listen socket 上有新连接到达，就调用我们刚刚注册的那个accept函数，并追加这个client socket到epoll中进行管理
  • 若发现其它 socket 上的读写请求，那就调用相应的读事件处理，和写事件处理。

3、为什么这么快？

基于内存

单线程避免了上下文竞争

io多路复用

高效的数据结构

4、采用单线程原因

  • 使用单线程模型是 Redis 的开发和维护更简单
  • 即使使用单线程模型也并发的处理多客户端的请求，主要使用的是多路复用IO
  • 对于 Redis 系统来说， 主要的性能瓶颈是内存或者网络带宽而并非 CPU。

5、引入多线程原因

内存问题比较好解决，因此Redis的瓶颈原因为网络IO

3、Redis持久化

Redis 共有三种数据持久化的方式：

  • AOF 日志：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里；
  • RDB 快照：将某一时刻的内存数据，以二进制的方式写入磁盘；
  • 混合持久化方式：Redis 4.0 新增的方式，集成了 AOF 和 RBD 的优点；

1、AOF

执行一条写命令，就把这个命令往aof文件里面写

1、为什么先执行命令，再把数据写入日志呢？

1、避免额外的开销

2、不会阻塞当前的写命令

缺点：

1、可能数据丢失，因为如果在执行完还没写入aof文件之间宕机，那就丢失

2、可能阻塞下一个命令

2、写回策略几种？

三种：Always everySec no，其实和mysql的binlog非常相似

第一种就是每次都同步

第二种就是每一秒都讲缓冲区里面的写会磁盘

第三种就是写到缓冲区里面，但是什么时候写到磁盘依赖于操作系统

3、AOF 日志过大，会触发什么机制？

aof重写机制：读取当前数据库中的所有键值对，对每一键值对用一条命令记录到「新的 AOF 文件」，等到全部记录完后，就将新的 AOF 文件替换掉现有的 AOF 文件。

aof是由后台子进程完成的。

这里有一个问题，就是如果aof重写过程中，主进程对内存进行了修改，怎么办？会触发写时复制

当aof重写过程中父进程修改内存，那我们执行一个写命令后，会写到aof缓冲区和aof重写缓冲区里面。当子进程完成重写，再去吧重写缓冲区里面的内容追加到aof文件

2、RDB快照

快照就是记录一瞬间的意思。记录的是当前的实际数据。

有两种启动的方式，一个save阻塞住线程，一种bgsave，后台线程。

所以rdb快照不会想aof那样开辟一个缓冲区用于记录新操作。这里rdb用的是全量快照数据。

如果在rdb快照的时候，主线程修改了内容，那么这时候会出发写保护中断，写时复制，原本的物理内存A，复制一份A‘，然后主线程修改A’就好了

3、混合持久化

混合持久化就是合并了aof和rdb的两个优点，rdb恢复数据块，aof丢失数据少

混合持久化，fork出来的子进程先去把共享内存以rdb的格式写入到aof文件里面，然后写入过程中主进程执行的命令会被记录到重写缓冲区里面，然后这些增量命令最后会以aof的格式增加到aof文件里面，所以aof文件里面前一部分时rdb，后一部分是aof。

4、Redis主从复制

主服务器可以进行读写操作，当发生写操作时自动将写操作同步给从服务器，而从服务器一般是只读，并接受主服务器同步过来写操作命令，然后执行这条命令。

主从服务器间的第一次同步是全量同步，分为三个阶段：

  • 第一阶段是建立链接、协商同步；

建立链接的时候会给主服务器发送psync，然后带上主服务器的runID和复制进度

主服务器收到后会返回一个FULLRESYNC响应，主服务器的runID和目前的复制进度

  • 第二阶段是主服务器同步数据给从服务器；

接着主服务器就执行bgsave生产rdb文件，发给从服务器，从服务器收到rdb文件，会清空当前自己的数据，然后载入rdb文件。并且把新增的写操作写入到replication buffer缓冲区

  • 第三阶段是主服务器发送新写操作命令给从服务器

把repliacation buffer缓冲区里面的写操作发给从服务器

第一次同步完成后，双方会维护一个TCP链接。

增量复制

网络断开又恢复后，从主从服务器会采用增量复制的方式继续同步，也就是只会把网络断开期间主服务器接收到的写操作命令，同步给从服务器。

主要有三个步骤：

  • 从服务器在恢复网络后，会发送 psync 命令给主服务器，此时的 psync 命令里的 offset 参数不是 -1（slave_repl_offset）
  • 主服务器收到该命令后，然后用 CONTINUE 响应命令告诉从服务器接下来采用增量复制的方式同步数据；
  • 主服务将主从服务器断线期间，所执行的写命令发送给从服务器，然后从服务器执行这些命令。

主服务器怎么知道要将哪些增量数据发送给从服务器呢？

  • repl_backlog_buffer，是一个「环形」缓冲区，用于主从服务器断连后，从中找到差异的数据。（主服务器进行命令传播时，不仅会将写命令发送给从服务器，还会将写命令写入到 repl_backlog_buffer 缓冲区里。）

  • replication offset，标记上面那个缓冲区的同步进度，主从服务器都有各自的偏移量，主服务器使用 master_repl_offset 来记录自己「写」到的位置，从服务器使用 slave_repl_offset 来记录自己「读」到的位置。

  • 如果判断出从服务器要读取的数据还在 repl_backlog_buffer 缓冲区里，那么主服务器将采用增量同步的方式；

  • 相反，如果判断出从服务器要读取的数据已经不存在 repl_backlog_buffer 缓冲区里，那么主服务器将采用全量同步的方式。

主从切换如何减少数据丢失？

主从切换有两种情况会数据丢失

  • 异步复制同步丢失。
  • 集群产生脑裂数据丢失。

第一种情况的解决方法：

Redis 配置里有一个参数 min-slaves-max-lag，表示一旦所有的从节点数据复制和同步的延迟都超过了 min-slaves-max-lag 定义的值，那么主节点就会拒绝接收任何请求。

第二种情况的解决方法：

当主节点发现「从节点下线的数量太多」，或者「网络延迟太大」的时候，那么主节点会禁止写操作，直接把错误返回给客户端。

  • min-slaves-to-write x，主节点必须要有至少 x 个从节点连接，如果小于这个数，主节点会禁止写数据。
  • min-slaves-max-lag x，主从数据复制和同步的延迟不能超过 x 秒，如果主从同步的延迟超过 x 秒，主节点会禁止写数据

如何如何应对主从数据不一致？

第一种方法，尽量保证主从节点间的网络连接状况良好，避免主从节点在不同的机房。

第二种方法，可以开发一个外部程序来监控主从节点间的复制进度。具体做法：得到主从的复制差距，如果过大，就不再和这个从节点进行数据同步

怎么判断 Redis 某个节点是否正常工作？

基本都是通过互相的 ping-pong 心态检测机制,

Redis 主节点默认每隔 10 秒对从节点发送 ping 命令

Redis 从节点每隔 1 秒上报自身当前的复制偏移量

5、Redis 过期删除与内存淘汰

1、Redis过期策略

Redis 使用的过期删除策略是「惰性删除+定期删除」这两种策略配和使用。

什么是惰性删除？

我不主动的删除，每当访问key的时候，我去判断一下是否过期，如果过期那么就删除该key

优点：占用比较少的资源，对cpu时间最友好

缺点：会导致缓存污染，浪费内存空间，对内存不友好

什么是定期删除策略？

就是定期的数据库里面随机抽取一定数量的key，并删除其中的过期的key

抽取策略？

1、从过期字典里面随机找20个

2、然后判断是否过期，并删除已经过期的key

3、如果过期的key超过了25%，那么就重复这个过程，如果没有，那就等待下一轮的抽查

优点：通过调整删除操作的时长的频率可以减少一些无用的占用

缺点：很难确定删除的时长和频率，太频繁对cpu不好，太少又和惰性删除一样。

1、Redis 持久化时，对过期键会如何处理的？

持久化有两种 aof和rdb

对于rdb，考虑生成和加载阶段

生成阶段：对于过期的key不写入rdb文件里面

加载阶段：这里分两种情况，一种是主服务器，另一种是从服务器。

​ 主服务器：加载阶段回去判断key是不是过期了，如果过期了那我就不加载这个

​ 从服务器：不管key是不是过期，全部加载。

aof：考虑aof写入阶段和重写阶段

写入阶段：如果这个key过期了，但是没有被删除，那么aof文件就会保留这个key，如果这个key被过期策略删除了，那么就会往aof文件里面追加一条del语句

重写阶段：重写阶段如果过期了，那么就直接删除，不会保留在新的aof文件里面

2、Redis 主从模式中，对过期键会如何处理？

主从模式中，从数据库完全依赖于主数据库，所以即使从库的key过期了，但是你查询，依旧能查到。只有主数据库在key到期的时候往aof文件里面增加del语句，同步到所有的从库，从库才会删除这个key。

2、Redis 内存淘汰

在 Redis 的运行内存达到了某个阀值，就会触发内存淘汰机制****

这个阀值就是我们设置的最大运行内存，此值在 Redis 的配置文件中可以找到，配置项为 maxmemory。

内存淘汰机制分为两类：一类是对有过期时间的内存淘汰机制；一类是对所有数据的淘汰机制。

对过期时间的内存淘汰机制可以分成四种：

1、volatile random

2、volatile lru 最近最少使用

3、volatile lfu 最近最不常用的

4、volatile ttl 最早时间

对所有数据的淘汰机制：

1、volatile random

2、volatile lru 最近最少使用

3、volatile lfu 最近最不常用的

LRU 算法和 LFU 算法有什么区别？

传统的lru算法依赖于一个链表，，数据被访问时候，链表中的节点会出现很多的移动操作，降低性能

LRU

Redis实现lru是在RedisObject里面新增一个字段，记录最后一次访问时间

进行淘汰的时候，随机选取五个，然后淘汰访问时间最早的

优点：节省空间，提高性能

缺点：无法解决缓存污染

LFU

lfu在RedisObject的基础上新增了一个字段，用于记录数据的访问频次。24位

高16位记录时间戳，后8位记录访问频次，默认5，会随着时间衰减

数据每次被访问的时候：

先做一个衰减操作，根据时间和上一次访问时间的差

然后做一次增加的操作，这个是基于一定的概率

6、Redis缓存设计

1、如何避免缓存雪崩？

缓存雪崩是指在同一时间大量的key同时过期了

解决方法：

这里关键就是同时和过期，所以解决方法如下：

1、打乱过期的时间

2、设置缓存不过期

2、缓存击穿

对于一个热点数据过期了，导致请求全部打到数据库

1、设置缓存不过期，后台异步的去更新

2、互斥锁，分布式锁的方法。让同一时间只有一个线程去访问

3、缓存穿透

访问的值既不在redis 也不在mysql，所以很可能是那种黑客攻击之类的，解决方法的话

1、非法请求的限制

2、空值或者默认值

3、布隆过滤器

布隆过滤器的主要原理是使用一组哈希函数，将元素映射成一组位数组中的索引位置。当要检查一个元素是否在集合中时，将该元素进行哈希处理，然后查看哈希值对应的位数组的值是否为1。如果哈希值对应的位数组的值都为1，那么这个元素可能在集合中，否则这个元素肯定不在集合中。

4、说说常见的缓存更新策略？

常见的缓存更新策略共有2种：

  • Cache Aside（旁路缓存）策略；

  • Read/Write Through（读穿 / 写穿）策略；

  • Write Back（写回）策略

旁路缓存策略

写策略的步骤：

  • 先更新数据库中的数据，再删除缓存中的数据。

读策略的步骤：

  • 如果读取的数据命中了缓存，则直接返回数据；
  • 如果读取的数据没有命中缓存，则从数据库中读取数据，然后将数据写入到缓存，并且返回给用户

读穿 / 写穿）策略

策略原则是应用程序只和缓存交互，不再和数据库交互，而是由缓存和数据库交互，相当于更新数据库的操作由缓存自己代理了。

1、Read Through 策略

  • 先查询缓存中数据是否存在，如果存在则直接返回，如果不存在，则由缓存组件负责从数据库查询数据，并将结果写入到缓存组件，最后缓存组件将数据返回给应用。

2、Write Through 策略

  • 当有数据更新的时候，先查询要写入的数据在缓存中是否已经存在：

    • 如果缓存中数据已经存在，则更新缓存中的数据，并且由缓存组件同步更新到数据库中，然后缓存组件告知应用程序更新完成。
    • 如果缓存中数据不存在，直接更新数据库，然后返回；

Write Back（写回）策略

写回在更新数据的时候，只更新缓存，同时将缓存数据设置为脏的，然后立马返回，并不会更新数据库。对于数据库的更新，会通过批量异步更新的方式进行。

实际上，Write Back（写回）策略也不能应用到我们常用的数据库和缓存的场景中，因为 Redis 并没有异步更新数据库的功能。

Write Back 是计算机体系结构中的设计，比如 CPU 的缓存、操作系统中文件系统的缓存都采用了 Write Back（写回）策略。

5、怎么保证数据一致性

redis和mysql的数据一致性其实有两类方式：一种是双写，一种是缓存删除。对于双写策略，两个请求并发更新同一条数据的时候，会出现数据不一致的情况，另外更新缓存的代价有时候是很高的，可能你1分钟内更新了100次，然后只访问了一次，那代价着实太大了。所以我们一般选择缓存删除的策略。

我们一般来说是先更新数据库，再去删除缓存，这个就是旁路缓存策略。虽然在理论上存在着 一个读请求读到旧值然后准备更新到redis里面的时候，有一个写请求更新数据库并且删除了缓存，这时候读请求才把旧值缓存到redis里面，这样出现redis和mysql的不一致。但是其实由于我们的缓存写入是远远快于数据库的写入，所以很难出现我一个请求更新完数据库删除完缓存了这时候，原先的读请求才写入缓存。为了保证第二个操作能成功，可以订阅binlog或者消息队列做删除。

对于并发引起的不一致性，那就用延迟双删
